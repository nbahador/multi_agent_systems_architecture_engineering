Multi-Agent System Template

Introduction to Multi-Agent Systems

This interactive textbook teaches you how to use the multi-agent system template to create distributed AI agent systems. The template enables you to build a master agent that coordinates multiple remote agent services.

Perfect for: Microservices architecture, distributed agent systems, service orchestration

Key Features
  • Coordinates multiple remote agent services
  • Uses RemoteA2aAgent to connect to agents on different servers
  • Supports both sequential and intelligent orchestration
  • Includes state tracking and callback mechanisms
  • Easy deployment as a web API

System Architecture

The multi-agent system follows an orchestrator pattern where a master agent coordinates multiple specialized remote agents.

# High-level architecture overview
from google.adk.agents.remote_a2a_agent import RemoteA2aAgent
from google.adk.agents.llm_agent import LlmAgent

# Create connections to remote services
service1 = RemoteA2aAgent(
  name="data_processor",
  description="Processes incoming data",
  agent_card="http://service1/.well-known/agent_card"
)

# Create master orchestrator
orchestrator = LlmAgent(
  name="master_coordinator",
  model="gemini-2.5-flash",
  sub_agents=[service1, service2, service3]
)

Architecture Components

The system consists of:

  • Master Orchestrator: Makes decisions about which agents to call
  • Remote Agents: Specialized services running on different servers
  • State Management: Tracks execution progress between calls
  • Callback System: Handles events during agent execution

The orchestrator analyzes each request and determines the optimal sequence of remote agents to invoke, passing data between them and handling any errors that occur.

Remote Agent Definitions

Remote agents are defined using the RemoteA2aAgent class, which connects to external agent services.

# --- FIRST REMOTE AGENT CONNECTION ---
# TODO: Replace with your first remote agent service
first_remote_agent = RemoteA2aAgent(
  # TODO: Give this remote agent a descriptive name

  # TODO: Write a brief description
  description="Handles initial data processing",
  # Build the agent card URL
  agent_card=(
    f"{FIRST_AGENT_URL}/{AGENT_CARD_WELL_KNOWN_PATH}"
  ),
)

Remote Agent Configuration

Each remote agent requires:

  • Name: A unique identifier for the agent
  • Description: Explains what the agent does
  • Agent Card URL: Points to the agent's metadata endpoint

The agent card contains information about the agent's capabilities, input/output formats, and other metadata that helps the orchestrator understand how to use the service.

The AGENT_CARD_WELL_KNOWN_PATH is typically "/.well-known/agent_card" - a standard location for agent metadata.

Callback Functions

Callback functions allow you to execute custom logic at different points in the agent execution process.

def save_last_action_callback(
  tool, # The tool/agent that just finished running
  args: Dict[str, Any], # Arguments that were passed to the tool
  tool_context: ToolContext, # Context object containing state and metadata
  tool_response: Dict[str, Any], # Response returned by the tool
) -> Optional[Dict[str, Any]]:
  """Callback that runs AFTER each remote agent finishes"""
  # Get the name of the agent that just finished
  agent_name = tool.name
  print(f"[Callback] After agent '{agent_name}' executed with args: {args}")

  # TODO: Customize what information you want to save
  print(f"[Callback] Saving last executed agent: {agent_name}")
  tool_context.state["last_action"] = agent_name

  # Return the original response
  return tool_response

Callbacks are powerful for implementing cross-cutting concerns like logging, monitoring, state tracking, and error handling across all agent interactions.

Callback Types
  • Before Tool Callback: Executes before an agent runs
  • After Tool Callback: Executes after an agent completes
  • Error Callback: Executes when an agent fails

You can use these callbacks to implement features like:

  • Execution timing and performance monitoring
  • State persistence between agent calls
  • Input validation and preprocessing
  • Error handling and fallback mechanisms
  • Logging and audit trails

Orchestrator Responsibilities

The master orchestrator has several key responsibilities:

  1. Request Analysis: Understands incoming user requests
  2. Service Selection: Chooses the appropriate remote agents
  3. Execution Order: Determines the optimal sequence
  4. State Management: Tracks execution progress
  5. Error Handling: Manages failures and retries

The orchestrator uses the LLM's reasoning capabilities to make intelligent decisions about which services to call based on the request content and current system state.

Orchestration Patterns

The template supports multiple orchestration patterns for different use cases.

Sequential Agent Pattern
# Use SequentialAgent for fixed-order execution
from google.adk.agents.sequential_agent import SequentialAgent

root_agent = SequentialAgent(
  name="sequential_service_coordinator",
  sub_agents=[
    first_remote_agent, # Always runs first
    second_remote_agent, # Always runs second
    third_remote_agent # Always runs third
  ],
  description="Calls remote services in a fixed sequence"
)

When to use: For predictable, linear workflows where the execution order is always the same.

Simple Proxy Pattern
# Use for simple forwarding to a single service
root_agent = RemoteA2aAgent(
  name="simple_proxy",
  description="Direct proxy to single remote service",
  agent_card=f"{FIRST_AGENT_URL}/{AGENT_CARD_WELL_KNOWN_PATH}"
)

When to use: When you need a consistent interface to a single remote service.

Server Deployment

The orchestrator can be deployed as a web API using the agent_to_a2a converter and uvicorn server.

# Import the converter that makes agents into web APIs
from agent_to_a2a import to_a2a

# TODO: Set your public URL for this orchestrator service
PUBLIC_URL = os.environ.get("PUBLIC_URL", "http://0.0.0.0:8080")

# This runs when you execute this file directly
if __name__ == "__main__":
  import uvicorn # Fast Python web server

  # TODO: Customize your server deployment settings
  SERVER_PORT = 8080
  SERVER_HOST = '0.0.0.0'

  # Convert your orchestrator agent into a web API
  a2a_app = to_a2a(
    root_agent, # Your orchestrator agent
    port=SERVER_PORT, # Port for this orchestrator service
    public_url=PUBLIC_URL # Public URL from environment
  )

  # Start the orchestrator web server
  uvicorn.run(a2a_app, host=SERVER_HOST, port=SERVER_PORT)

The agent is converted to an A2A (Agent-to-Agent) web API, which standardizes the interface for agent communication and makes it accessible over HTTP.

    • Test complete workflows
